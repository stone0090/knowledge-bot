# 个人知识库搭建方案

## 零、分期计划

| 期号 | 范围 | 状态 |
|------|------|------|
| **一期** | URL 投喂（Jina Reader）、纯文本投喂、LLM 按 **SCHEMA** 编译为实体页/概念页、**ECS Vault 存储（md + git）**、**飞书云盘只读镜像**（Wiki 产物 docx）、`/查` 检索（ripgrep 本地扫，含 queries 回填）、本地联调 + ECS 部署 | 联调中 |

| **三期** | 多源抓取扩展（YouTube/B站字幕 yt-dlp、推特 xreach、RSS、Reddit/Exa 语义搜索）、Wiki lint（矛盾/过时/孤儿/缺索引）、周报 | 待启动 |

## 一、核心决策

| 维度 | 选型 | 理由 |
|------|------|------|
| 存储 | **ECS 本地 Vault（纯 md）+ Git 版本化** | 数据主权，零 SaaS 锁定；Obsidian / VS Code 直接编辑；打包 tar 即可迁移；每次编译自动 commit → 天然的幻觉回滚网 |
| Git 中央节点 | **ECS 自建 bare 仓库**（`/opt/vault-bare.git`） | 不经任何第三方托管（含 GitHub / GitLab / Gitee）；SSH 协议加密；每个端都是完整副本 |
| AI 能力 | 阿里云百炼（Qwen 系列） | 中文友好，Coding Plan 费用可控 |
| IM 入口 | 飞书自建机器人 | 官方 SDK 完善，手机端即时投喂最方便 |
| 阅读端 | 桌面 Obsidian · iOS Working Copy + Obsidian · Android MGit + Obsidian | 跨端 git 同步、离线可读 |
| 飞书角色 | **只读镜像**：仅镜像 Wiki 编译产物为 docx 到云盘 | 手机没装 Obsidian 时的备用阅读入口；**best-effort，失败仅告警**；不承担主存储 / 不承担检索 |
| 方法论 | LLM Wiki 三层架构 + 红绿灯原则 | 对标 Karpathy LLM Wiki：Raw 只读、SCHEMA 当契约、人机分三档避免幻觉污染 |

## 二、系统架构（四层）

```
[用户]
  ↓ 发送链接 / 文本 / 文件
[飞书机器人]                 ← 入口层：接收消息、下发回复卡片
  ↓ webhook
[FastAPI 后端（ECS）]        ← 处理层
  ├─ 抓取层                  详见 §五「URL 抓取分层策略」
  ├─ 编译层                  按 SCHEMA.md 抽实体/概念 → 生成 Wiki 页 → 刷新 index/log
  ├─ LLM                     阿里云百炼 · OpenAI 兼容协议
  ├─ Vault 写入              写 md + frontmatter → git commit/push
  └─ 飞书 API                仅：IM 收发 + 云盘 docx 镜像写入（best-effort）
  ↓
[ECS 存储层]                 ← 真相源（详见 §三）
  ├─ /opt/vault/             工作副本（FastAPI 在这里读写）
  │   ├─ Raw/{articles,notes,files,videos}/
  │   ├─ Wiki/{entities,concepts,comparisons,queries}/
  │   ├─ Wiki/{index.md,log.md}
  │   └─ SCHEMA.md
  └─ /opt/vault-bare.git/    裸仓库（中央节点，所有端 SSH clone 这里）
  ↓ 镜像（best-effort）              ↓ SSH git clone / pull / push
[飞书云盘「知识库-镜像」]           [你的设备]
  只读阅读端：                       ├─ 电脑：Obsidian + obsidian-git
  仅镜像 Wiki 编译产物 docx；         ├─ iOS：Working Copy + Obsidian
  失败不阻断主流程                   └─ Android：MGit + Obsidian
```

## 三、知识组织：LLM Wiki 三层架构

参考 Karpathy LLM Wiki 范式（详见 [docs/外部参考/01-知识库构建.md](外部参考/01-知识库构建.md)），知识资产全部落在 ECS 的 `/opt/vault/` 目录（纯 md 文件 + git 版本化），按三层组织：

### 第一层 · Raw（原始来源，**LLM 只读不改**）

按**来源类型**归档，便于后续 lint 按类型扫查：

- `/opt/vault/Raw/articles/`   URL 抓取正文（一期主力）
- `/opt/vault/Raw/notes/`      用户直发的文本笔记（一期）
- `/opt/vault/Raw/files/`      PDF / PPT / Excel / 图片（二期启用）
- `/opt/vault/Raw/videos/`     YouTube / B 站字幕（三期启用）

### 第二层 · Wiki（LLM 编译产物，自动维护）
- `/opt/vault/Wiki/entities/`      实体页：工具 / 人 / 项目（一期）
- `/opt/vault/Wiki/concepts/`      概念页：一个概念一页（一期）
- `/opt/vault/Wiki/comparisons/`   对比页：A vs B（二期）
- `/opt/vault/Wiki/queries/`       查询回填页：`/查` 出的好答案归档（一期）
- `/opt/vault/Wiki/index.md`       总目录：导航入口
- `/opt/vault/Wiki/log.md`         操作日志：每次编译追加

### 第三层 · SCHEMA（人机契约，你和 AI 共同维护）
- `/opt/vault/SCHEMA.md`           定义：页面命名、模板、标签体系、新材料处理流程、`/查` 回答格式

### 检索索引层 · md frontmatter + 本地扫描

每个 md 文件头部以 YAML frontmatter 承载元数据（`title / domain / tags / summary / source_url / created_at`）。检索走本地：

- **一期**：`ripgrep` 扫 `/opt/vault/Wiki/**/*.md` 的 frontmatter + 正文关键词，毫秒级返回
- **二期（可选）**：SQLite FTS5 建索引文件（`/opt/vault/.index.db`），支持中文分词与权重排序

> 不再依赖飞书多维表格：元数据随 md 走，零外部依赖、零同步成本、零字段漂移。

### 飞书云盘镜像（只读阅读端）
- `/飞书云盘/知识库-镜像/`   **仅镜像 Wiki 编译产物**转 docx（Raw / SCHEMA 不镜像），每次 ingest 完成后 best-effort 推一份；失败仅打告警，不阻断主流程

> **差异对比**：传统 PARA / Zettelkasten 以「分类」为组织主线；LLM Wiki 以「编译动作」为组织主线，分类只是副产物——更适合 LLM 自动维护 index 与 log。

## 四、知识编译与维护（红绿灯原则）

Karpathy 在 LLM Wiki gist 里的金句：**人类放弃 Wiki 不是因为不会读，而是没人愿意持续维护索引与交叉引用**。LLM 正是那个不知疲倦的维护员；但为避免幻觉污染 Wiki，按红绿灯三档划分人机分工：

| 等级 | 策略 | 示例 |
|------|------|------|
| 🟢 绿灯 | 全托管给 LLM | 摘要生成、索引更新、链接补全、格式调整、孤儿页检查 |
| 🟡 黄灯 | 人机共审 | 矛盾裁决、概念合并、过时内容作废（每周 lint 过一遍） |
| 🔴 红灯 | 绝不外包 | 核心事实写入、价值判断、最终签字 |

### 定期 lint

让 LLM 扫出四类问题：

1. **互相打架的论点**：新页面推翻旧论点但旧页面没改
2. **过时内容**：几个月前的判断是否还成立
3. **孤儿页**：没有任何地方链接过来的页面
4. **索引缺口**：高频提及但没有专页的概念

分期落地：

- **一期**：不跑 lint，页面量少，靠编译时追加 log.md 即可
- **二期**：上 `/lint` 手动指令（M12 前置）
- **三期**：APScheduler 周期触发 + 飞书周报卡片推送（M12）

## 五、URL 抓取分层策略

参考 [docs/外部参考/02-URL读取工具.md](外部参考/02-URL读取工具.md) 提出的“不造轮子，只整合”哲学，按**页面类型 + 反爬难度**分层：

| 分层 | 抓取工具 | 覆盖 | 期次 |
|------|---------|------|------|
| L1 · 纯 HTML 正文 | **Jina AI Reader** `https://r.jina.ai/{url}` | 90% 通用网页、GitHub 公开仓库、RSS 条目 | ✅ 一期 |
| L2 · 结构化文件 | **markitdown** | PDF / PPT / Excel / Word / 图片 OCR | 🟡 二期 |
| L3 · 视频字幕 | **yt-dlp** + `--write-auto-subs` | YouTube / B站 / 1800+ 视频站 | 🔵 三期 |
| L4 · 登录态平台 | **xreach**（推特）、Cookie 注入 | Twitter / 小红书 / 抖音 / LinkedIn / Boss直聘 | 🔵 三期 |
| L5 · JS 渲染兜底 | **Playwright** 浏览器渲染 | 必须 JS 才能出内容的单页应用 | 🟠 按需 |
| ⛔ 死胡同 | — | **微信公众号**（反爬强度业界公认无解，绕过属灰产，项目不做） | 永不 |

### 一期为什么只要 Jina Reader

- 一条 `curl` 即可拿回干净 Markdown，不吃内存、不启动浏览器
- 无需注册即可使用，个人投喂节奏下成本可忽略
- 90% 的技术文章、资讯站、博客类 URL 都能解析
- 二期之后再逐层叠加，保持「轻装上路、按需扩展」

### 抓取失败的降级策略

| 失败场景 | 行为 | 是否告警 |
|---------|------|---------|
| 返回空 markdown | 记录为「抓取空」卡片，保留原 URL 待人工处理 | 仅日志 |
| 403 / 反爬 | 明确回复「该域名需要 L4 登录态抓取，三期再支持」 | 飞书卡片提示 |
| 微信公众号域名 | 前置识别 `mp.weixin.qq.com` → 建议用户“复制正文发给我” | 飞书卡片提示 |
| 超时 | 重试 1 次 → 仍失败则降级为“仅保留 URL + 标题” | 仅日志 |

## 六、两条核心工作流

### 流程 A：投喂（Ingest）

1. 用户在飞书里 @机器人 或私聊发送 URL / 文本（一期）/ 文件（二期）
2. 机器人识别类型 → 走 §五 抓取分层 → 拿到纯文本
3. 原文按来源类型写入 `/opt/vault/Raw/{articles|notes|files|videos}/`（**此后 LLM 永不修改**）
4. LLM 按 `SCHEMA.md` 编译：
   - 抽出实体（工具/人/项目）→ 生成/更新 `/opt/vault/Wiki/entities/*.md`
   - 抽出概念 → 生成/更新 `/opt/vault/Wiki/concepts/*.md`
   - 元数据写入 md 文件头部 YAML frontmatter（title/domain/tags/summary/source_url/created_at）
   - 若与既有页面矛盾 → 在页面头部标注「⚠️ 冲突待裁决」
   - 追加 `/opt/vault/Wiki/index.md` 与 `/opt/vault/Wiki/log.md` 条目
5. `git add . && git commit -m "ingest: <title>" && git push`（推向 `/opt/vault-bare.git`；失败仅告警不阻断）
6. **飞书镜像（best-effort）**：把本轮新增/更新的 Wiki md 转 docx 推到 `飞书云盘/知识库-镜像/`；失败仅日志告警
7. 飞书卡片回复：标题 / 摘要 / vault 路径 / 飞书镜像 docx 链接（镜像成功时附带）

### 流程 B：检索（Query）

1. 用户发 `/查 关键词` 或自然语言问题
2. 机器人用 `ripgrep` 扫 `/opt/vault/Wiki/**/*.md`（**不扫 Raw**，查询始终命中编译产物）召回 Top-K 候选页
3. LLM 基于候选页 frontmatter + 正文生成带引用的回答
4. 答案质量达标时回填到 `/opt/vault/Wiki/queries/YYYYMMDD-{question-slug}.md`（通过卡片「保存到 Wiki」按钮触发），同样触发 git commit/push 与飞书镜像
5. 飞书蓝色卡片回复：答案 / 引用来源（Wiki 页 vault 路径 + 原文 URL）
